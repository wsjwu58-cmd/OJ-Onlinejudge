## 1. 功能拓展

### 1.1 连接原单体项目的已有功能

当前微服务项目依赖 `oj-project/` 作为功能参考源，需要排查是否所有功能已迁移完毕：

| 待排查模块 | 当前状态 | 建议 |
|-----------|---------|------|
| 用户签到/积分体系 | `user_attendance` 表已存在 | 需要前端对应对接和后端接口 |
| 代码查重系统 | 未见相关模块 | 原项目若有需补充 |
| 题库难度标签 | `problems` 表缺少难度字段索引 | 影响筛选性能 |
| 排行榜/排名系统 | Contest 服务有基本排名 | 可增加全局排名 |

### 1.2 功能完善建议

- **管理端 AI 接口**：当前 AI 服务仅有用户端路由，管理端可能需要统计分析
- **判题结果追溯**：增加判题详情历史记录（输入/输出捕获），方便用户 Debug
- **题目推荐系统**：基于用户提交历史，利用 AI 服务做题目推荐
- **数据大盘**：管理端增加系统运营大盘（日活、提交量、AC率趋势）

## 2. 数据库与持久层优化 (MySQL/MyBatis-Plus)

### 2.1 事务管理 — **高优先级**

**现状**：整个微服务项目中 `@Transactional` 注解**零使用**，所有多表操作均无事务保护。

受影响的典型场景：

| 服务 | 方法 | 风险 |
|------|------|------|
| ProblemService | `problemSave()` | Insert Problem + Insert ProblemTypesRel，中途失败导致脏数据 |
| ProblemService | `updateProblem()` | 删除+重新插入类型关联无原子性 |
| ContestService | `saveContest()` | 创建比赛+插入题目关联无事务 |
| ContestService | `update()` | 更新比赛+删除/重插关联无事务 |
| GroupService | `saveGroup()` / `update()` | 题组操作无事务 |
| UserContestService | `joinContest()` | 检查+插入+Redis ZAdd，涉及跨资源操作 |

**立即改进措施**：
- 单服务多表操作：添加 `@Transactional(rollbackFor = Exception.class)`
- 跨服务+数据库操作：需要 Seata 分布式事务（见第11节）

### 2.2 N+1 查询问题 — **高优先级**

**DB 层面 N+1**（同服务内）：

| 位置 | 问题描述 | 影响 |
|------|---------|------|
| `ProblemServiceImpl.pageQuery()` | 每页 20 条数据，每条再单独查询类型列表 | 21 次查询 |
| `ProblemServiceImpl.queryType()` | 同上 | 21 次查询 |
| `ContestServiceImpl.updateContestStatus()` | `selectList(null)` 全表加载后逐条更新 | 内存+查询风暴 |

**解决方案**：
```java
// 批量映射方案
List<Long> problemIds = page.getRecords().stream().map(Problem::getId).toList();
Map<Long, List<ProblemType>> typeMap = problemTypeService
    .list(new LambdaQueryWrapper<ProblemTypeRel>().in(ProblemTypeRel::getProblemId, problemIds))
    .stream().collect(Collectors.groupingBy(...));
```

使用 MyBatis-Plus 的 `selectBatchIds()` 批量查询替代循环。

**Feign 层面 N+1**（跨服务）— **更严重**：

| 位置 | 问题 | 每次页面请求的远程调用数 |
|------|------|------------------------|
| `SubmissionServiceImpl.getSubmission()` | 逐条调用 `getProblemById()` | 1 + N 次 |
| `SubmissionServiceImpl.pageQuery()` | 每行提交记录调 2 次 Feign | 1 + 2N 次 |
| `UserContestServiceImpl.pageContest()` | 每个比赛逐条查参与人数 | 1 + N 次 |
| `ContestServiceImpl.deleteId()` | 逐条校验题目状态 | N 次 |
| `ProblemServiceImpl.deleteAll()` | 逐条校验比赛引用 | N 次 |

**解决方案**：
```java
// 批量收集 ID，一次性调用
Set<Long> userIds = submissions.stream().map(Submission::getUserId).collect(toSet());
Set<Long> problemIds = submissions.stream().map(Submission::getProblemId).collect(toSet());
Map<Long, User> userMap = userClient.getUsersByIds(userIds);
Map<Long, Problem> problemMap = problemClient.getProblemsByIds(problemIds);
// 再回填
```

### 2.3 索引优化

**缺失的关键索引**：

| 表 | 建议添加索引 | 理由 |
|----|------------|------|
| `problems` | `idx_status (status)` | 按状态筛选是高频操作 |
| `problems` | `idx_difficulty (difficulty)` | 按难度筛选 |

| `user` | `idx_role (role)` | 管理端按角色筛选 |
| `user` | `idx_status (status)` | 账户状态筛选 |

| `contests` | `idx_status (status)` | 按状态筛选比赛 |
| `contests` | `idx_start_time (start_time)` | 按时间范围查询 |

### 2.4 Druid 连接池调优

**当前配置过于简陋**：

```yaml
# 当前 — 仅配置了连接数
spring.datasource.druid:
  initial-size: 20
  min-idle: 20
  max-active: 100
  max-wait: 5000
```

**建议补充**：

```yaml
spring.datasource.druid:
  # 原有配置
  initial-size: 20
  min-idle: 20
  max-active: 100
  max-wait: 5000
  # 建议新增
  test-on-borrow: true              # 借用连接时验证
  test-while-idle: true             # 空闲连接验证
  validation-query: SELECT 1        # 验证 SQL
  time-between-eviction-runs-millis: 60000   # 60s 清理一次
  min-evictable-idle-time-millis: 300000     # 5min 最小空闲时间
  remove-abandoned: true            # 回收泄漏连接
  remove-abandoned-timeout: 180     # 超时 180s
  log-abandoned: true               # 记录泄漏日志
  # Druid 过滤器建议开启
  filters: stat,wall               # 统计 + SQL 防火墙
  filter:
    stat:
      slow-sql-millis: 1000        # 慢 SQL 阈值 1s
      log-slow-sql: true
    wall:
      enabled: true
  # StatViewServlet 监控面板
  stat-view-servlet:
    enabled: true
    login-username: admin
    login-password: ${druid.password}
```

**注意**：当前 `initial-size: 20` × 4 个服务 = 80 个初始连接。生产环境需根据实际负载调整，建议初始值降低到 5-10。

### 2.5 批量操作优化

**现状**：`saveBatch()`, `updateBatch()` 从未使用，批量插入走的是多条单条 INSERT。

**改进**：
- 使用 `saveBatch(Collection, batchSize)` 实现真正的 JDBC batch 插入
- 对 `contestProblemMapper.insert(contestProblems)` 等批量插入启用 batch 优化

### 2.6 危险查询修复

| 问题 | 位置 | 修复 |
|------|------|------|
| `select * from problems` 无 LIMIT | `ProblemMapper.selectAll()` | 改为分页查询或添加 LIMIT |
| XML 中引用旧库名 `exercise.xxx` | `UserMapper.xml`, `ProblemMapper.xml` | 更新为当前库名 |

## 3. 缓存与分布式协调优化 (Redis)

### 3.1 当前 Redis 使用情况

| 特性 | 状态 | 使用位置 |
|------|------|---------|
| 用户登录态缓存 | ✅ 已使用 | Hash: `login:user:{token}` |
| 排行榜实时排名 | ✅ 已使用 | ZSet: `contest_rank:{contestId}` |
| 判题速率限制 | ✅ 已使用 | Lua 脚本 + ZSet 滑动窗口 |
| 判题结果原子更新 | ✅ 已使用 | Lua 脚本状态机 |
| 缓存注解集成 | ❌ 未使用 | @Cacheable / @CachePut / @CacheEvict |
| 分布式锁 (Redisson) | ❌ 未使用 | setnx / Redisson Lock |
| 布隆过滤器 | ❌ 未使用 | 防缓存穿透 |
| Redis Stream | ❌ 未使用 | 轻量消息队列 |
| 发布/订阅 | ❌ 未使用 | 实时通知 |

### 3.2 缓存穿透/击穿/雪崩防护 — **高优先级**

**当前完全未覆盖**。建议方案：

```java
@Service
public class ProblemCacheService {
    // 1. 热点题目缓存 + 空值缓存防穿透
    @Cacheable(value = "problem", key = "#id", unless = "#result == null")
    public Problem getProblemById(Long id) { ... }
    
    // 2. 热点题目防击穿（分布式锁）
    public Problem getProblemWithLock(Long id) {
        String lockKey = "lock:problem:" + id;
        RLock lock = redissonClient.getLock(lockKey);
        try {
            if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
                // 查询 DB 并回设缓存
            }
        } finally {
            lock.unlock();
        }
    }
    
    // 3. 随机过期时间防雪崩
    @Cacheable(value = "problems", key = "#page")
    @CachePut(value = "problems", key = "#page")
    public PageResult<ProblemVO> pageQuery(Integer page, Integer size) {
        // TTL = baseTTL + Random.nextInt(300) 秒
    }
}
```

### 3.3 缓存策略建议

| 数据类型 | 缓存策略 | TTL | 更新触发 |
|---------|---------|-----|---------|
| 题目详情 (热点) | Cache-Aside | 1h ± random(5min) | 题目更新时删除 |
| 题目列表 (分页) | Cache-Aside | 10min | 新题添加/修改时删除第一页 |
| 用户信息 | Cache-Aside | 30min | 用户更新时删除 |
| 比赛信息 | Write-Through | 比赛期间永久 | 比赛状态变更时更新 |
| 排行榜 (Top100) | Write-Through (ZSet) | 比赛结束后 24h | 每次 AC 提交时更新 |

### 3.4 分布式锁

使用 Redisson 替代手工 Lua + Redis：

```java
// 判题去重（替代手工 Lua 脚本中的速率限制逻辑）
RLock lock = redissonClient.getLock("judge:lock:" + userId + ":" + problemId);
if (lock.tryLock(0, 10, TimeUnit.SECONDS)) {
    try {
        // 判题逻辑
    } finally {
        lock.unlock();
    }
}
```

### 3.5 Redis Stream 替代 RocketMQ（可选）

对于非关键通知类消息（如比赛即将开始的提醒、系统公告），可用 Redis Stream 实现轻量级异步通知，降低 RocketMQ 负载。

## 4. 消息队列深度运用 (RocketMQ)

### 4.1 当前使用情况

仅 `oj-judge-service` 使用 RocketMQ，存在两条消息流：
```
judge-task-topic        → JudgeTaskConsumer       → Judge0 判题
database-update-topic   → DatabaseUpdateConsumer  → 结果持久化
```

### 4.2 可扩展的消息场景

| 场景 | 说明 | 优先级 |
|------|------|--------|
| 题目状态变更通知 | 题目新增/更新时通知缓存和搜索引擎 | 中 |
| 比赛开始/结束事件 | 触发排名结算、通知下发 | 中 |
| 用户注册事件 | 异步初始化用户数据（默认头像/设置） | 低 |
| 代码查重异步任务 | 提交后触发相似度检测 | 高 |
| 磁盘文件清理 | 删除题目时清理 OSS 图片和附件 | 中 |
| 数据统计异步更新 | 提交后更新全局统计数据 | 中 |

### 4.3 高级特性引入

| 特性 | 当前状态 | 建议 |
|------|---------|------|
| **Tag 过滤** | 未使用 | 判题语言区分：`judge-task-topic:java`, `judge-task-topic:cpp` |
| **顺序消息** | 未使用 | 同一用户提交需保证 FIFO 排序 |
| **事务消息** | 未使用 | 配合 Seata 做分布式事务最终一致性 |
| **批量消息** | 未使用 | 比赛结束后批量结算排名 |
| **消息追踪** | 未使用 | 接入 SkyWalking Trace 上下文 |
| **ACL 鉴权** | 未使用 | 生产环境必须开启 |

### 4.4 DQL（死信队列）增强

**现状**：Dead Letter Consumer 仅 `log.error()` 记录，无恢复和告警。

**改进**：
```java
@RocketMQMessageListener(topic = "%DLQ%judge-task-consumer-group", ...)
public class JudgeTaskDeadLetterConsumer {
    @Override
    public void onMessage(MessageExt msg) {
        // 1. 记录到死信表 (MySQL/task_dead_letter)
        // 2. 超出阈值时企业微信/钉钉告警
        // 3. 管理员可手动重试或丢弃
    }
}
```

### 4.5 消息幂等性

**潜在风险**：RocketMQ 重试可能导致消息重复消费。

**建议**：在 `JudgeTaskConsumer` 和 `DatabaseUpdateConsumer` 中添加幂等校验：
```java
// 基于 submissionId 的 Redis 去重
String idempotentKey = "msg:dedup:" + msgId;
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent(idempotentKey, "1", Duration.ofHours(24));
if (!Boolean.TRUE.equals(success)) return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
```

## 5. 流控与熔断体系 (Sentinel)

### 5.1 现状评估

Sentinel 依赖已引入所有服务，但**零规则配置、零注解使用**。仅 Feign 的 FallbackFactory 有最基础的降级返回。

### 5.2 必须配置的 Sentinel 规则

**流量控制规则**（FlowRule）：

| 资源 | 规则 | QPS 阈值 | 理由 |
|------|------|---------|------|
| `judge:submit` | QPS 流控 | 50/s | 防止 Judge0 被打爆 |
| `user:login` | QPS 流控 | 20/s | 防止暴力破解 |
| `admin:*` | QPS 流控 | 100/s | 管理端保护 |
| `problem:list` | QPS 流控 | 200/s | 热点接口保护 |

**熔断降级规则**（DegradeRule）：

| 资源 | 策略 | 阈值 | 降级方式 |
|------|------|------|---------|
| `problemClient` | 慢调用比例 | RT > 1000ms | 返回缓存数据 |
| `judge0:submit` | 异常比例 | > 50% | 返回"系统繁忙" |
| `userClient` | 异常数 | > 10/min | FallbackFactory |

**热点参数规则**（ParamFlowRule）：

```java
@SentinelResource(value = "problemDetail", blockHandler = "problemDetailBlock")
public Problem getProblemById(Long problemId) {
    // 对高频访问的题目 ID 限流
}

// Sentinel 热点规则
ParamFlowRule rule = new ParamFlowRule("problemDetail")
    .setParamIdx(0)           // 第0个参数 problemId
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setCount(100);           // 单题 QPS 上限
```

### 5.3 网关层限流

```yaml
# Spring Cloud Gateway + Redis RateLimiter
spring:
  cloud:
    gateway:
      routes:
        - id: judge-user
          uri: lb://oj-judge-service
          predicates:
            - Path=/api/user/judge/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter:
                  replenishRate: 10      # 每秒令牌数
                  burstCapacity: 20      # 突发容量
                  key-resolver: "#{@userKeyResolver}"
```

当前未连接 Sentinel Dashboard。建议：
```yaml
spring:
  cloud:

      transport:
        dashboard: 192.168.141.129:8858   # Sentinel Dashboard 地址
```

## 6. 高并发与性能治理

### 6.1 当前瓶颈评估

| 组件 | 风险点 | 预估瓶颈 |
|------|--------|---------|
| Judge0 | 单实例，HTTP 同步调用 | 并发判题上限约 100 QPS |
| 数据库 | 单实例，连接池 400 | 高并发下连接数耗尽 |
| Redis | 单实例 | 内存上限 |
| RocketMQ | 单 Broker | 消息堆积 |
| Gateway | 单实例 | 无自我保护机制 |

### 6.2 性能优化措施

#### 6.2.1 Judge0 调用优化

```java
// 当前：同步 HTTP 调用 Judge0，通过 CompletableFuture 并发
// 改进：使用连接池 + 请求合并
HttpClient httpClient = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2)       // HTTP/2 多路复用
    .connectTimeout(Duration.ofSeconds(3))
    .build();

// 或考虑批量提交至 Judge0 的 batch API
```

#### 6.2.2 连接池复用

所有 Feign 客户端应配置连接池，避免每次请求新建连接：
```yaml
spring:
  cloud:
    openfeign:
      httpclient:
        hc5:
          enabled: true     # 使用 Apache HttpClient 5 连接池
      client:
        config:
          default:
            connect-timeout: 3000
            read-timeout: 10000
```

#### 6.2.3 异步化改造

**现状**：仅判题流程使用 RocketMQ 异步。其他同步调用可异步化：

```java
// 比赛开始/结束后的非核心操作异步化
@Async
public void onContestStart(Long contestId) {
    // 1. 刷新参赛用户缓存
    // 2. 发送通知
    // 3. 初始化排行榜
}
```

#### 6.2.4 数据库读写分离

随规模增长考虑：
- **现阶段**：单库够用，先优化索引和查询
- **下一阶段**：MySQL 主从 + ShardingSphere 读写分离
- **未来**：按业务垂直分库（用户库、题库、比赛库、判题库已初步分离）

### 6.3 高并发场景专项治理

| 场景 | 问题 | 方案 |
|------|------|------|
| 比赛开始瞬间流量尖峰 | 大量用户同时进入比赛页 | Redis 预热 + CDN 静态资源 + 页面限流 |
| 热门题目大量提交 | Judge0 过载 | 提交队列 + 动态优先级 + 题目维度限流 |
| 排行榜实时计算 | ZSet 大 Key (> 10MB) | 拆分为 Top100 + 全量异步计算 |
| 比赛结束排名结算 | 瞬时大量写操作 | RocketMQ 削峰填谷 + 分批结算 |

## 7. 安全加固

### 7.1 密码安全 — **最高优先级**

**严重问题**：
- 密码以**明文存储**在数据库
- Admin 登录**不校验密码**
- 默认密码硬编码 `"123456"`

**修复方案**：
```java
// oj-user-service 添加 Spring Security Crypto
implementation 'org.springframework.security:spring-security-crypto'

@Configuration
public class SecurityConfig {
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();  // BCrypt
    }
}

// 注册时加密
user.setPasswordHash(passwordEncoder.encode(rawPassword));

// 登录时校验
if (!passwordEncoder.matches(rawPassword, user.getPasswordHash())) {
    throw new LoginFailedException("密码错误");
}
```

### 7.2 接口安全

| 问题 | 修复 |
|------|------|
| CORS 允许所有来源 (`*`) | 限定白名单域名 |
| 无 CSRF Token | 管理端添加 CSRF 防护 |
| 无 XSS 过滤 | Gateway 添加 XSS Filter |
| 密码尝试无限制 | 登录接口添加验证码 + 锁定机制（连续失败 5 次锁定 15 分钟） |
| 管理员密码不校验 | **立即修复** |

### 7.3 Token 安全

**当前 Admin JWT**：使用硬编码明文密钥 `itcast-sky-take-out-admin-secret-key-2024`。

**改进**：

## 8. 可观测性与运维

### 8.1 日志体系

**现状**：依赖 Spring Boot 默认 logback，仅 YAML 中配置了日志级别。无滚动策略、无文件输出、无 ELK 集成。

**建议**：创建 `logback-spring.xml`：
```xml
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/${spring.application.name}.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
        <fileNamePattern>logs/${spring.application.name}-%d{yyyy-MM-dd}.%i.log</fileNamePattern>
        <maxHistory>30</maxHistory>
        <totalSizeCap>10GB</totalSizeCap>
    </rollingPolicy>
    <encoder>
        <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} [%X{traceId}] - %msg%n</pattern>
    </encoder>
</appender>
```

关键：添加 `[%X{traceId}]` 配合 SkyWalking 做链路追踪。

### 8.2 SkyWalking 配置

SkyWalking Agent 已通过 `.gitignore` 排除（暗示在服务器上有部署），建议将 Agent 配置模板（不含密钥）提交到仓库 `skywalking-agent/config/agent.config.template`。

### 8.3 Prometheus + Grafana

**现状**：Micrometer + Prometheus 指标已完善（特别是 JudgeMetrics 的 10 个自定义 Timer），但缺少 Grafana 看板。

**建议**：
1. 创建 Grafana Dashboard JSON 模板放入 `doc/grafana/` 目录
2. 关键监控面板：
   - 各服务 QPS / 延迟 (P50/P95/P99)
   - 判题成功率 / 平均耗时
   - RocketMQ 消息堆积量
   - 数据库连接池使用率
   - JVM 内存 / GC 统计

### 8.4 Gateway 健康检查

**现状**：Gateway 是唯一没有 Actuator 的服务。

**改进**：Gateway 添加 `spring-boot-starter-actuator` 并暴露 `/actuator/health`，方便 K8s/Docker 存活探测。

### 8.5 告警规则

## 11. 分布式事务 (Seata)

### 11.1 现状

`doc/Seata分布式事务实施方案.md` 已有详细设计文档（520 行），但代码中**零实现**。

### 11.2 需要分布式事务的场景

| 场景 | 涉及服务 | 风险 |
|------|---------|------|
| 判题结果写入 | Judge → User → Problem → Contest | 中途失败导致用户解题数不一致 |
| 比赛报名 | Contest + Redis(ZSet) | 名额超卖 |
| 题目删除 | Problem → Contest (引用检查) | 已被比赛引用的题目被删除 |
| 用户解题数更新 | Judge → Problem (AC 计数) + User (解题数) | 数据不一致 |

### 11.3 实施建议

按照文档中的 Seata AT 模式推进：
1. 部署 Seata Server (Docker, 192.168.141.129:7091)
2. 创建 `seata` 数据库和 `undo_log` 表
3. Nacos 中配置 `seata.properties`
4. 涉及跨服务写操作的方法添加 `@GlobalTransactional`
5. Feign 调用链添加 XID 传播（`FeignConfig` 中已有 `BaseContext` 机制可复用）

## 12. 架构层面优化

### 12.1 服务边界调整

| 当前问题 | 建议 |
|---------|------|
| Problem Service 承载了题目+题组+知识库，职责过重 | 考虑拆分 oj-knowledge-service |
| 前端两套代码有大量重复组件 | 抽取 oj-frontend-common 组件库 |
| Contest 的排行结算依赖 Feign 同步调用 | 改为 RocketMQ 事件驱动 |

### 12.2 服务注册发现健壮性

**当前**：所有服务依赖 Nacos，无本地缓存降级。

**建议**：开启 Nacos 客户端本地快照：
```yaml
spring:
  cloud:
    nacos:
      discovery:
        naming-load-cache-at-start: true
        watch:
          enabled: true
```

### 12.3 配置中心管理

- 将 `application.yml` 中可动态变更的配置（如限流阈值、开关）迁移至 Nacos Config
- 本地 `bootstrap.yml` 仅保留连接 nacos 的最小配置

### 12.4 数据一致性保障

| 维度 | 当前 | 建议 |
|------|------|------|
| 用户解题数 | 判题成功后才更新 | 考虑最终一致性 + 补偿任务 |
| 题目 AC 率 | 判题成功后才更新 | 同上 |
| 比赛排名 | 实时 Redis ZSet | 添加 MySQL 定期快照做对账 |
| 缓存-数据库一致性 | 无缓存 | 采用 Cache-Aside + 延迟双删 |
